# Distributed Single Scan on Spark (DSSS)
### A Modern Reproduction Study on Spark 3.5.4

This repository contains the code and artifacts for our reproduction of the
**Distributed Single Scan on Spark (DSSS)** algorithm originally proposed by
Rochd & Hafidi (2025). DSSS is a single-pass frequent itemset mining
algorithm built on Spark's distributed, in-memory execution model.

We validate the original claims on two real-world datasets — **HIGGS** and
**OTTO** — and benchmark DSSS against Spark MLlib's **FP-Growth**.

---

## Authors

- **Original Research:** Yassir Rochd & Imad Hafidi (2025)
- **Reproduction Study:** Hamdah Iqbal, Asad Ali

---

## Highlights

- **50–70× speedup** over Spark MLlib's FP-Growth on HIGGS at $k \le 2$.
- **Near-linear scalability** from 2 to 16 local cores.
- **Stable runtime (~12–15 s)** on dense OTTO transactions where FP-Growth
  frequently runs out of heap.
- **Validated correctness:** DSSS singleton/pair counts match FP-Growth and
  a SQL ground-truth baseline exactly.

## Setup

1. **Clone or unzip the project.**
2. **Edit the paths** at the top of each script to match your machine. The
   scripts currently assume:
   - HIGGS root: `/mnt/d/DS3/`
   - OTTO parquet: `/home/hi/DS3_work/otto_transactions.parquet`
   - Spark scratch: `/mnt/d/DS3/tmp/` and `/home/hi/DS3_work/tmp/`

   Search-and-replace these to point to your own `data/`, `results/`, and
   `plots/` folders.
3. **Place the raw inputs:**
   - `data/raw/higgs_subset_header.csv` — the HIGGS subset (100k rows) with
     a header row and a `label` column (excluded from features).
   - `data/raw/train.csv` — the OTTO training file (`id`, 93 `feat_*`, and
     `target` columns; `id` and `target` are excluded from features).

## How to Run

The pipeline runs in four phases. Each script can be executed with
`spark-submit` or just `python` if PySpark is on your `PYTHONPATH`.

### 1. Prepare transactions

Both datasets are discretized into 10 quantile bins per feature using
Spark's `approxQuantile`, then encoded as transactional items
(e.g., `f0_q3`, `feat_6_q0`).

```bash
python prepare_transactions_higgs_subset.py
python prepare_transactions_otto.py
```

Outputs Snappy-compressed Parquet under `data/processed/` and a small
stats file (`dataset_stats_otto.txt`).

### 2. Run DSSS

DSSS is implemented with `mapPartitions` for local hash-table counting
and `reduceByKey` for the global merge — a Spark-3.x-safe replacement for
the original paper's broadcast variables. Candidate generation is capped
at $k \le 2$ for feasibility on dense datasets.

```bash
# HIGGS
python run_dsss_higgs_k2.py             # prints results to stdout
python run_dsss_higgs_k2_savecsv.py     # also writes results/higgs_dsss_k2_summary.csv

# OTTO
python run_dsss_otto_full.py
python run_dsss_otto_full_savecsv.py    # writes results/otto_dsss_k2_runtime.csv
```

Each run sweeps `minSup` over `[0.001, 0.002, 0.003, 0.004, 0.005]`.

### 3. Run baselines

```bash
# FP-Growth baseline (toggle via env var)
DATASET=higgs python run_fpgrowth_k2_sweep.py
DATASET=otto  python run_fpgrowth_k2_sweep.py

# SQL ground truth (singletons + exact pairs on a 20k HIGGS sample)
python exact_pairs_baseline.py
```

> **Note:** FP-Growth frequently throws Java heap OOM at very low support
> thresholds, particularly on OTTO. This is a real limitation of the
> baseline, not a configuration error.

### 4. Scalability + plots

```bash
python run_scaling_higgs_localcores.py    # sweeps local[2/4/8/16]
python make_plots.py                      # builds all four figures
```

`run_scaling_higgs_localcores.py` starts a fresh `SparkSession` per core
count so the `local[N]` master takes effect cleanly.

## Implementation Notes

- **Why $k \le 2$?** OTTO has 93 dense features per transaction; full
  candidate enumeration explodes combinatorially. We cap at singletons +
  pairs, which is enough to validate correctness against FP-Growth and SQL
  ground truth without exhausting memory.
- **Why not broadcast variables?** Spark 3.x broadcast variables are
  immutable, so the original DSSS pattern of mutating a shared hash table
  no longer works. `mapPartitions + reduceByKey` is the equivalent
  Spark-safe shape and is what the scripts implement.
- **FP-Growth `numPartitions`.** We deliberately do *not* set
  `numPartitions` on `FPGrowth`; setting it triggers a known degenerate
  partitioning bug at low support.
- **Determinism.** `exact_pairs_baseline.py` uses `zipWithIndex` to take
  the first 20,000 rows so the SQL ground truth is reproducible across
  runs.

---

## Limitations

- Single-node Spark only — cluster-level shuffle and network costs are
  not measured.
- Candidate generation capped at $k \le 2$.
- FP-Growth comparisons are bounded by its OOM behaviour at low support.
- Validated on two datasets only (HIGGS, OTTO).
